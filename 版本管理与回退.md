## 一、文档概述

### 1.1 目的

为 train-departure-diary 项目建立系统化的版本迭代策略和回退机制，确保：

- 版本号规范统一
- 多环境隔离（dev/staging/production）
- 支持蓝绿部署和热回退
- 数据库变更安全可控
- 多端版本策略（H5/小程序/App）

### 1.2 现状分析

| 项目     | 当前状态                         | 问题                |
| -------- | -------------------------------- | ------------------- |
| 前端版本 | package.json: 1.0.0, 代码: 1.0.0 | 与后端 1.1.0 不一致 |
| 后端版本 | 代码: 1.1.0, /api/version: 1.1.0 | 与前端不一致        |
| 环境配置 | 只有 production (3000)           | 无 staging，无蓝绿  |
| CI/CD    | 无                               | 纯手动部署          |
| 数据库   | 远程阿里云 + 本地同步            | 无版本化迁移脚本    |
| 回退方式 | 手动覆盖代码                     | 耗时且风险高        |

## 二、语义化版本规范

### 2.1 版本号格式

```
Major.Minor.Patch[-PreRelease]

示例:
1.1.0       - 正式版本
1.2.0-beta  - 预发布版本
1.1.1       - 热修复版本
```

### 2.2 递增规则

| 级别               | 触发条件                      | 示例          |
| ------------------ | ----------------------------- | ------------- |
| **Major** (主版本) | 破坏性变更，不兼容的 API 变更 | 1.x.x → 2.0.0 |
| **Minor** (次版本) | 向下兼容的新功能              | 1.1.x → 1.2.0 |
| **Patch** (修订号) | 向下兼容的问题修复            | 1.1.x → 1.1.1 |

### 2.3 当前版本对齐

| 文件                | 当前版本 | 目标版本 |
| ------------------- | -------- | -------- |
| package.json (前端) | 1.0.0    | 1.1.0    |
| server/package.json | 1.0.0    | 1.1.0    |
| server/index.js     | 1.1.0    | 1.1.0    |
| manifest.json       | -        | 1.1.0    |

## 四、部署架构

### 4.1 蓝绿部署

**原理**: 维护两套后端环境（绿=当前生产，蓝=备用版本），通过 nginx 一键切换。

```
                    ┌── 3000 (Green - 当前生产)
用户请求 ──▶ nginx ──┤
                    └── 3001 (Blue - 备用版本)
```

**切换流程**:

```
紧急回退:
1. 执行 rollback.sh v1.0.0
2. nginx upstream 切换到 3001
3. 耗时: < 5 秒
```

### 4.2 目录结构

```
/root/train-departure-diary/
├── server-v1.0.0/          # 稳定旧版本 (端口 3001)
├── server-v1.1.0/          # 当前版本 (端口 3000)
├── server-staging/          # 预发布版本 (端口 3002)
└── server-current -> server-v1.1.0/  # 软链接
```

### 4.3 前端多版本策略

```
CDN 存储结构:
/static/
├── v1.0.0/
│   ├── js/app.js
│   └── css/app.css
├── v1.1.0/
│   ├── js/app.js
│   └── css/app.css
└── current -> v1.1.0/
```

## 六、数据库变更策略

### 6.1 核心原则

| 原则           | 说明                   |
| -------------- | ---------------------- |
| 始终向前兼容   | 新代码同时读写新旧字段 |
| 保留旧列 30 天 | 确认无引用后再删除     |
| 变更前先备份   | mysqldump 完整备份     |
| 使用事务包裹   | DDL 操作尽量在事务中   |

### 6.2 SQL 版本化结构

```
sql/
├── schema-mysql.sql              # 基础建表
├── add_updated_at.sql            # 历史迁移
├── migrations/
│   ├── 001_add_profit_v2.sql    # 迁移脚本
│   └── 002_xxx.sql
└── rollback/
    └── 001_rollback_profit_v2.sql  # 对应回滚
```

### 6.3 新增字段流程

```sql
-- 阶段 1: 添加新列（向前兼容）
ALTER TABLE departures ADD COLUMN profit_v2 DOUBLE DEFAULT NULL;

-- 阶段 2: 代码部署 - 同时读写新旧字段

-- 阶段 3: 数据迁移（后台异步，确认稳定后）
UPDATE departures SET profit_v2 = profit WHERE profit_v2 IS NULL;

-- 阶段 4: 确认稳定后（30天后），清理旧数据
-- DELETE FROM departures WHERE profit IS NOT NULL AND profit_v2 IS NULL;

-- 阶段 5: 最终删除旧列
-- ALTER TABLE departures DROP COLUMN profit;
```

### 6.4 备份策略

| 类型       | 频率        | 保留时间 |
| ---------- | ----------- | -------- |
| 全量备份   | 每周        | 90 天    |
| 增量备份   | 每日        | 30 天    |
| 变更前备份 | 每次 DDL 前 | 永久     |

## 七、前端多端版本策略

### 7.1 平台差异

| 平台       | 发布方式   | 回退难度 | 推荐策略                    |
| ---------- | ---------- | -------- | --------------------------- |
| H5         | CDN 多版本 | 简单     | nginx 路由切换              |
| 微信小程序 | 代码审核   | 依赖微信 | 灰度比例 + 回退到上一正式版 |
| App        | wgt 热更新 | 中等     | 强制更新 + 增量包           |

### 7.2 版本检测机制

```typescript
// App.vue 或 store/settings.ts
async function checkVersion() {
  const res = await fetch("/api/v1/version");
  const { version, minAppVersion } = await res.json();

  const currentVersion = "1.1.0"; // 构建时注入
  const isCompatible = compareVersion(currentVersion, minAppVersion) >= 0;

  if (!isCompatible) {
    uni.showModal({
      title: "版本过低",
      content: "请更新到最新版本后使用",
      showCancel: false,
    });
  }
}
```

## 八、回退流程

### 8.1 蓝绿回退（推荐）

```bash
# 服务器上执行
cd /root/train-departure-diary
./rollback.sh v1.0.0

# 脚本内容:
# 1. 停止当前服务
pm2 stop server-current

# 2. 切换软链接到旧版本
ln -sf server-v1.0.0 server-current

# 3. 重启服务
pm2 restart server-current

# 4. 验证
curl http://localhost:3000/api/v1/version
```

### 8.2 前端回退

```bash
# H5 回退 - 切换 CDN 路由
# nginx 配置
location /static {
    alias /var/www/static/v1.0.0/;  # 切换到旧版本
}
```

### 8.3 数据库回退（不推荐）

```sql
-- 仅在严重数据问题时考虑
-- 1. 停止后端服务
-- 2. 恢复备份
mysql -u root -p train_departure_diary < backup_20260408.sql
-- 3. 重启后端
-- 4. 通知受影响用户
```
