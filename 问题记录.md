# 问题记录

## 问题记录表

| 问题 | 原因 | 修改 | 影响范围 | 反思 | 日期 |
| ---- | ---- | ---- | -------- | ---- | ---- |
| 鸡场列表和员工列表数据重复 | 统一使用的 store 管理，并且数据是追加操作 | 改为直接替换数据 | 鸡场列表、员工列表 | store 管理数据时应注意是替换还是追加操作，避免数据累积 | 2026-04-06 |
| 分页处理后数据重复 | 进入发车记录时也会触发接口请求，进入鸡场列表、员工列表时也会触发请求，而且使用了追加操作，导致出现了数据重复 | 对接口调用时机进行了修改 | 发车记录、鸡场管理、员工管理 | 多页面共享 store 时要注意接口调用时机和数据更新策略，避免重复请求导致数据叠加 | 2026-04-06 |
| 直接进入某些页面数据为空 | 只在进入鸡场管理和员工管理时触发接口请求，直接进入其他页面，没有触发接口请求，导致了空数据现象 | 将填写发车记录页面、结账页面、单次结账页面自行调用鸡场列表接口和员工列表接口并在自身页面进行维护 | 发车记录、结账、单次结账页面 | 各页面应自主管理数据加载，不依赖其他页面的数据初始化，确保直接进入时数据可用 | 2026-04-07 |
| 发车记录表单中的「鸡场金额明细」缺失 | 计算逻辑中使用了 merchantStore.getMerchantById 去 store 中获取对应的鸡场信息，但是没有进过鸡场管理，此时 store 中的数据还是空的，导致不能正常获取到鸡场信息，导致相关计算出错 | 将 store 中的鸡场管理和员工管理移入各自页面后删除，其他需要使用鸡场数据和员工数据的页面自行调用接口并维护数据 | 发车记录表单 | 计算时应确保依赖数据已加载完成，避免依赖未经初始化的 store 数据，页面应自行管理其所需的数据生命周期 | 2026-04-07 |
| 发车记录保存时触发多次重复的 PUT 请求 | saveRecords 设计为遍历所有 records，云端化后未优化 | 待修改：saveRecords 应只保存被修改的单条记录 | 发车记录保存 | 存储方式变更时需要同步优化相关逻辑。最初 saveRecords 为 localStorage 设计（同步一次性写入），迁移到云端 MySQL 后仍保留遍历逻辑，导致每次只修改一条记录却 PUT 所有记录 | 2026-04-08 |
